⚡ Background Workers

Background work is handled by BullMQ running in a separate Node.js process (npm run worker). All workers connect to the same Redis instance and MongoDB database as the web app.

Queue NameWorkerTriggered ByDoes What
content-generation contentGeneration.ts POST /api/posts Calls AI provider, generates post content, saves as draft
auto-publish autoPublish.ts Post approval or AUTO_PUBLISH_DELAY Publishes approved post to LinkedIn, creates PublishedPost record
scheduled-publish autoPublish.ts ScheduledPost delay job Publishes a scheduled post at the specified date/time
campaign-runner campaign.ts BullMQ cron (campaign schedule) Generates post for campaign, routes to auto-publish or email approval
notification notification.ts Other workers Saves in-app notification, optionally sends SMTP email
token-refresh tokenRefresh.ts Cron (every 6h) Refreshes LinkedIn OAuth access tokens before expiry

Campaign Cron Scheduling

When a campaign is activated, a repeating BullMQ job is added with a cron expression computed from the schedule config:

Schedule ConfigCron Expression
Weekly, Monday + Wednesday, 9:00 AM0 9 * * 1,3
Weekly, Friday, 6:30 PM30 18 * * 5
Monthly, 1st + 15th, 9:00 AM0 9 1,15 * *

When a campaign is paused or deleted, the repeating job is removed via queue.removeRepeatableByKey(key). The job key is stored on the Campaign document.

🤖 AI Provider Module

All AI providers implement a common IAIProviderClient interface:

interface IAIProviderClient {
  chat(messages: Message[]): Promise<string>
  getModels(): Promise<string[]>
  testConnection(): Promise<boolean>
}

Providers are instantiated via factory:

const client = createAIProvider(provider, config)
// provider: 'openai' | 'claude' | 'gemini' | 'ollama' | 'openrouter'
// config: { apiKey, ollamaEndpoint, defaultModel, generationSettings }

🔒 Security

Authentication

  • Session-based authentication via NextAuth.js with HTTP-only cookies
  • Passwords hashed with bcrypt (salt rounds: 12)
  • JWT sessions signed with NEXTAUTH_SECRET
  • Role-based access: user vs admin

Encryption

  • All API keys (OpenAI, Claude, etc.) encrypted with AES-256-GCM before MongoDB storage
  • LinkedIn OAuth tokens encrypted with the same key
  • Encryption key is never stored — must be provided via environment variable
  • Each encrypted value uses a unique random IV (initialisation vector)

Rate Limiting

  • Configurable rate limiting via RATE_LIMIT_MAX and RATE_LIMIT_WINDOW_MS
  • Applied at the API route level

Data Isolation

  • All database queries are scoped to userId from the session
  • Admin routes verify role === "admin" on every request
  • LinkedIn accounts are never shared between users
