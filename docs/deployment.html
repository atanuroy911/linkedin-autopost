Deployment

LinkedIn AI Publisher is deployed via Docker Compose. The deploy.sh script handles building images and starting containers in the correct order.

Docker Compose Deployment

1. Prepare Environment File

# On your server
git clone https://github.com/your-username/linkedin-autopost.git
cd linkedin-autopost

# Create production env file
cp .env.example .env.production
nano .env.production  # Fill in all values
🔴

Set a strong, unique NEXTAUTH_SECRET and ENCRYPTION_KEY. These must never change after first deployment or encrypted data becomes unreadable.

2. Run deploy.sh

chmod +x deploy.sh
./deploy.sh

The deploy script does the following:

  1. Starts a Redis container (if not already running)
  2. Stops and removes old web and worker containers
  3. Builds new Docker images for both services
  4. Starts linkedin-autopost-web on port 4000
  5. Starts linkedin-autopost-worker (BullMQ background jobs)

3. Seed the Admin Account

# Run seed inside the web container
docker exec linkedin-autopost-web node scripts/seed.js
# or if npm run seed is configured:
docker exec linkedin-autopost-web npm run seed

docker-compose.yml Overview

version: '3.8'

services:
  mongodb:
    image: mongo:latest
    ports: ["27017:27017"]
    volumes: [mongodb_data:/data/db]

  redis:
    image: redis:alpine
    ports: ["6379:6379"]

  web:                   # Next.js App
    build: { target: web }
    ports: ["4000:3000"]
    env_file: [.env.production]
    depends_on: [mongodb, redis]

  worker:                # BullMQ Background Jobs
    build: { target: worker }
    env_file: [.env.production]
    depends_on: [mongodb, redis]

Setting Up a Reverse Proxy (Nginx)

Expose the app on port 80/443 via Nginx:

📄 /etc/nginx/sites-available/linkedinpost 
server {
    listen 80;
    server_name linkedinpost.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name linkedinpost.example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    location / {
        proxy_pass         http://localhost:4000;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_cache_bypass $http_upgrade;
    }
}
# SSL with Let's Encrypt
certbot --nginx -d linkedinpost.example.com

# Enable site and reload
ln -s /etc/nginx/sites-available/linkedinpost /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx

Update LinkedIn Redirect URI

After deploying to production, update your LinkedIn Developer App with the production redirect URI:

https://linkedinpost.example.com/api/linkedin/callback

Also update LINKEDIN_REDIRECT_URI and NEXT_PUBLIC_APP_URL in .env.production, then redeploy.

Backup & Recovery

# Backup MongoDB
docker exec linkedin-autopost-mongo \
  mongodump --db linkedin-autopost --out /data/backup
docker cp linkedin-autopost-mongo:/data/backup ./mongo-backup-$(date +%Y%m%d)

# Restore MongoDB
docker cp ./mongo-backup linkedin-autopost-mongo:/data/backup
docker exec linkedin-autopost-mongo \
  mongorestore --db linkedin-autopost /data/backup/linkedin-autopost
⚠️

Back up your .env.production file securely. Losing the ENCRYPTION_KEY means all stored API keys and LinkedIn tokens become unrecoverable.

Production Checklist

  • Strong, unique NEXTAUTH_SECRET (32+ chars)
  • Strong, unique ENCRYPTION_KEY (64 hex chars via openssl rand -hex 32)
  • NODE_ENV=production set
  • MongoDB with authentication enabled (not open to public)
  • Redis not exposed to public internet
  • HTTPS with valid SSL certificate
  • LinkedIn OAuth redirect URI updated to production URL
  • NEXT_PUBLIC_APP_URL set to production HTTPS URL
  • SMTP configured for email notifications
  • Regular database backups scheduled
  • .env.production not committed to git
